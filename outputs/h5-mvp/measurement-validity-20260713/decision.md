# AI 曝光体检测量有效性判定

## 1. 一句话结论

**C：红色。当前四模型离线 API 采样不足以代表 DeepSeek、通义、腾讯元宝、豆包消费端的真实品牌认知与推荐体验；应停止把现有结果解释成消费端曝光或排名，优先验证真实消费端搜索与引用采集路线。**

这是对当前测量方法的判定，不是否定“品牌认知体检”本身的用户需求。

## 2. 实际测试规模

### API（已确认事实）

- 4 个外部实体 × 4 条统一提示词 × 4 个 provider，共 64 个主样本。
- 63 个成功回答；1 个失败：豆包 API 的追觅 P2 超时。
- 0 次模型 fallback；隔离运行器关闭了 SDK 重试。
- 64/64 均为当前产品实际离线采样方式：没有联网搜索、搜索工具或引用检索。

### 消费端（已确认事实）

- 2026-07-13 在同一日内完成 64 次网页端独立新会话，保存 64 份页面证据和 64 张截图。
- 57 条取得可评分的最终回答；7 条为生成卡住、请求失败、停留在工具状态或只回显提示词。
- DeepSeek：16 次，16 条可评分；默认快速模式，未手动开启智能搜索。
- 通义：16 次，15 条可评分；可见模式为 Qwen3.7-千问，支付宝 P3 生成卡住后停止。
- 腾讯元宝：16 次，13 条可评分；默认“元宝”模式，追觅 P1/P3 停在“正在搜索商品”，P4 请求失败。
- 豆包：16 次，13 条可评分；默认快速模式，支付宝 P2/P3、负对照 P2 未形成最终回答。
- 57 条可评分回答中，33 条显示自动联网；四个平台是否搜索取决于平台和问题，并非固定行为。

## 3. 数据边界

- `raw/api/` 是当前产品四个 provider 的离线 API 回答。
- `raw/consumer/` 是四个网页消费端的回答、搜索痕迹、链接、异常状态和截图。
- API 未联网不是遗漏实验变量，而是当前真实测量方法：现有采样代码没有给 provider 配置搜索工具、联网参数或引用检索。若现在临时开启，会测到另一种方法，不能回答“当前产品是否有效”。
- 消费端单次回答不等于稳定排名；本轮只比较单次默认体验的方向和证据。

## 4. 三个主要指标

### 指标 1：品牌认知方向一致性

评分判断：按 P1 与 P3 都需同方向的严格口径，两个目标产品共 8 个实体—平台配对中：

| 配对 | 结果 | 关键依据 |
|---|---|---|
| DeepSeek × flomo | 冲突 | API P1 明确无法确认；消费端联网后正确识别。 |
| 通义 × flomo | 一致 | P1/P3 双方均正确识别产品方向；消费端另有公司主体错误。 |
| Hy3 API × 腾讯元宝 flomo | 冲突 | API P1 仅部分认识；消费端正确认识。 |
| 豆包 × flomo | 一致 | P1/P3 双方均正确识别产品方向。 |
| DeepSeek × 追觅 | 冲突 | API P1/P3 不认识；消费端联网后正确识别和比较。 |
| 通义 × 追觅 | 冲突 | API 不认识并在 P3 断言产品不存在；消费端正确识别。 |
| Hy3 API × 腾讯元宝追觅 | 无法评价 | 消费端 P1/P3 均未形成最终回答。 |
| 豆包 × 追觅 | 冲突 | API P1 不认识、P3 错误；消费端联网后正确识别产品方向。 |

结果为 **2/8 一致、5/8 冲突、1/8 无法评价**。即使把 Hy3 对 flomo 的“部分认识”宽松并入认识，也只有 3/8，不改变红色结论。

### 指标 2：无品牌自然提及一致性

- flomo：2/4 配对一致。DeepSeek 双方均未提及；Hy3 API 与腾讯元宝均提及（第 3 / 第 2）；通义和豆包均出现 API 未提及、消费端提及。
- 追觅 X50 Ultra：3/3 可比较配对一致，均未提及；豆包 API P2 超时，不能算一致或冲突。豆包消费端提到的是 X50 S Pro，不计作 X50 Ultra。
- 合计 **5/7 可比较配对一致**。但 flomo 只有 2/4，说明消费端搜索与推荐策略可直接改变是否入榜。

### 指标 3：事实与来源一致性

- API：0/64 联网。两个目标产品的 8 条 P4 中，仅 2 条被评为至少有一个有效引用；并出现把离线内容标成“已核验”、错误链接和错误存在性结论。
- 消费端：57 条有效回答中 33 条自动联网。两个目标产品有 7 条可用 P4，均至少有一个可支持部分主张的链接，但只有 5/7 含官方来源；搜索仍会带来错主体、错规格和无关来源。
- 最强反例是追觅：DeepSeek、通义、豆包 API 都未能稳定识别，三个消费端却能正确识别；其中 DeepSeek、豆包消费端自动搜索，通义甚至在未显示搜索时也正确识别。

评分判断：来源与联网机制的差异足以推翻主要诊断，API 不能代表消费端搜索曝光。

## 5. 正负对照是否有效

- 正对照支付宝：P1 在 4 个 API 和 4 个消费端均正确识别；API P2 四家均主动提及，消费端 3 条有效 P2 也均主动提及。说明提示词和核心识别模式总体可用。
- 负对照栖栈清单：4 个消费端 P1/P3 都没有把它当作已确认产品；DeepSeek、Qwen、Hy3 API 也安全处理。但豆包 API P3 将其编造成真实国内行李工具，并虚构受众、下载渠道和榜单依据。

结论：**正对照有效；负对照在消费端有效，但 API 质量护栏失败一次。正负对照并非均正常，因此不能判绿色。**

## 6. 三个典型一致案例

1. 通义 × flomo P1/P3：API 与消费端都将其识别为轻量碎片/卡片笔记工具，并与 Notion 的结构化工作空间方向区分开。
2. 豆包 × flomo P1/P3：双方对核心产品定义和适用人群方向一致。
3. Hy3 API × 腾讯元宝 flomo P2：无品牌提示下双方都主动提及 flomo，位置分别为第 3 和第 2。

## 7. 三个典型冲突案例

1. DeepSeek × 追觅 P1/P3：离线 API 不认识；消费端自动搜索后正确识别并完成比较。
2. 通义 × 追觅 P1/P3：API 怀疑或断言型号不存在；消费端在 P1/P3 未显示搜索仍正确识别。
3. 豆包 × 追觅 P1/P3：API 不认识并编错规格；消费端自动搜索后正确识别产品方向，虽仍混合了不同地区版本规格。

## 8. 严重编造和无效引用清单

### 严重编造 / 核心事实反转（评分判断，共 6 条）

1. API 通义 flomo P4：把未联网内容标成已核验，并给出错误或不支持主张的官方/App Store 链接。
2. API 通义 追觅 P3：错误断言 X50 Ultra 与 X8 PRO OMNI 不存在或未量产。
3. API 通义 追觅 P4：再次错误断言 X50 Ultra 未发布。
4. API 豆包 追觅 P3：混淆或编造吸力、滚刷、拖地、导航和避障规格。
5. API 豆包 栖栈清单 P3：为虚构实体编造产品类别、用户、下载渠道和榜单依据。
6. 消费端通义 flomo P1：自动搜索后仍把运营公司“上海仙蒂”错写成“上海鲜淡”。

### 无效或不足以支持主张的引用

- Qwen API flomo P4 的 App Store ID 与当前官方基准不一致，且离线回答错误标为已核验。
- 通义消费端 flomo P1/P3 混入与笔记产品事实无关的手游排行页面。
- 通义消费端 flomo P4 含格式异常的豆瓣跳转链接。
- Qwen API 追觅 P4 给出的品牌或电商域名不能支持“X50 Ultra 不存在”的主张。
- DeepSeek 消费端追觅 P4 称未找到官方产品页，但公开官方产品页实际存在。

## 9. 判定

**红色，对应决策 C。**

触发条件：

- 目标产品严格一致配对只有 2/8，不超过红色阈值 4/8；
- 多个平台对追觅是否存在、做什么得出相反结论；
- 负对照 API 出现严重编造；
- 共发现 6 条严重编造或核心事实反转；
- API 与消费端的联网、来源和推荐结果存在实质差异。

这不是 5/8 边界情形，不触发仅复测冲突提示词的规则。

## 10. 对产品的直接影响

以下为基于证据的产品判断：

- **产品名称**：不得再把现有“AI曝光体检”理解为四个消费端的曝光体检；若名称暂不改，界面和报告必须显著声明是 API 样本，不代表消费端搜索或排名。
- **销售口径**：停止使用“代表 DeepSeek、通义、元宝、豆包真实推荐/曝光”的表述；API 未提及不能解释为消费端未曝光。
- **技术路线**：当前关键缺口不是再扩大离线模型调用，而是真实消费端搜索过程、引用和默认模式的可追溯证据。单独给 API 开联网只能验证新的 API 测量法，仍不能自动等同消费端。
- **推广节奏**：暂停以核心定位做公开扩量、付费承诺或销售验证；保留研究原型，等待测量路线验证。

## 11. 尚未验证的事项

- 消费端单次结果在不同日期、账号、地区、移动 App 和网页端的稳定性。
- 平台可见模式与后台实际模型版本的对应关系。
- 消费端搜索结果、引用顺序和推荐位置的重复性。
- 合规、可持续地采集真实消费端搜索与引用证据是否可行。
- 四家 provider 的联网 API 能否改善事实和来源质量；即使改善，也仍需另证其与消费端的一致性。

## 12. 下一步只允许一个动作

**只做 1 份内部正式详细报告 dry run：选择 1 个真实外部实体，使用四消费端独立新会话采集、公开来源核验和人工复核完整走通交付链路；不扩大 API 调用，不公开推广。**

## 13. 2026-07-14 小样本生死复盘（仅内部）

### 数据是否足以做决定

- 128 条评分记录完整覆盖 64 个 API 单元和 64 个消费端尝试；消费端 57 条可评分、7 条异常，64 份 JSON 与 64 张截图均保留。
- 独立复算确认：目标实体品牌认知严格一致 2/8、冲突 5/8、不可比较 1/8；无品牌自然提及 5/7 一致；严重编造 6 条，其中 API 5 条、消费端 1 条。
- 这仍是每个实体—提示词—平台只有一次回答的小样本，不能用于估计稳定排名、跨日波动或商业转化；评分由单一评价流程完成，也没有盲评和双人复核。
- 但主要结论距离边界很远，并且存在“API 不认识或断言不存在、消费端正确识别”的核心事实反转；正对照支付宝有效，负对照又检出 API 编造。因此现有样本足以否定“离线 API 可以代表消费端默认体验”这一原假设。

### 生死判断

- **原核心假设：死亡。** 当前离线 API 结果不能被解释为 DeepSeek、通义、腾讯元宝或豆包消费端的真实曝光、搜索结果或推荐排名。
- **用户问题与产品机会：保留。** 企业确实需要知道模型如何认识品牌、公开网页是否有可信证据，以及真实消费端在单次默认体验中如何回答；本轮没有否定这个需求。
- **免费 H5：有条件保留。** 只作为“多模型品牌认知与公开证据快检”，输出 API 样本和候选公开来源；必须显著说明不代表消费端实时搜索或稳定排名。
- **正式详细报告：有条件保留。** 必须加入真实消费端采集、完整原始证据、来源核验和人工复核，不能由 API 自动报告直接升级而来。
- **30 秒并行 SLA：仅内部工程条件。** 单次整链路 27.535 秒返回完整报告结构，但只成功 37/40 个模型回答；该数字不作为对外口径，也不改变测量有效性的红色判定。

### 最终结论

**正式判定仍为 C（红色）：当前测量方法不足以支持“AI 曝光体检代表四个消费端真实体验”的核心定位。产品不整项终止，但必须杀掉该代表性承诺，按“免费 H5 做 API 品牌认知与候选公开证据、正式报告做真实消费端采集与人工核验”的两层路线继续。**

该判断可用于内部产品去留与路线决策；由于样本为单日、单次、网页端且评分未双盲复核，不应对外宣传为行业普遍规律或平台稳定表现。

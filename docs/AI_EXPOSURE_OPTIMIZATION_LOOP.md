## 总目标

在 `/Users/qzt/Developer/geo-lab` 内按顺序完成四个独立 Goal。任何时刻只允许一个 Goal 处于 `in_progress`。每个 Goal 完成实现后必须先通过自己的验收门，再更新 checkpoint 并进入下一项；不得为了继续循环而弱化测试、隐藏失败或把未验证结果写成完成。

1. G1：实体判定器与采样前来源核验。
2. G2：AI曝光体检官方内容与结构化证据。

机器状态：`docs/ai-exposure-optimization-loop-state.json`

## 共享执行规则

- 开始或恢复时先完整读取 `AGENTS.md`、`README.md`、`PROJECT_CONTEXT.md`、`TODO.md`、`DECISIONS.md`、`BUG_NOTES.md`、本文件和机器状态文件。
- 当前权威项目路径只有 `/Users/qzt/Developer/geo-lab`，不得修改冰箱小雷达或其他项目。
- 不生成子代理，不创建并行线程；四个 Goal 串行执行。
- 默认自动权限仅为 L1/L2：可读取、运行本地命令、修改本项目代码/测试/文档/fixture/输出证据。
- 未获得本轮单独确认前，不 commit、push、部署、发布、修改生产 runtime、发送生产诊断 POST、修改 API Key/IP 白名单/云权限/DNS、访问需要登录的后台、产生可能付费的模型或搜索调用。
- 不引入登录、数据库、支付、Docker、队列、微服务或新 provider。
- 不使用脚本批量删除文件；不删除或覆盖现有用户文件；不使用 `git reset --hard`。
- 不打印或保存 API Key、Token、Cookie、Authorization header、账号信息、真实源 IP 或生产表单内容。
- 保留现有可信度合同：无已核验来源不出精确总分；品牌复述不等于自然推荐；冲突和 provider 失败必须展示；所有导出使用同一可信度数据。
- 修改代码必须先建立能复现问题的失败测试，再做最小修复。
- 连续两轮没有新证据时，状态改为 `waiting_for_user`，记录尝试、证据、风险和需要用户判断的问题。

## 通用验收门

每个 Goal 的专属验收通过后，还必须完成：

- `npm run typecheck`
- `npm test`
- `npm run build`
- `git diff --check`
- 相关自动化/fixture 回归全部通过，未删除、跳过或弱化旧测试。
- 若有用户界面变化，完成 390×844 与 1440×1000 的真实浏览器验证，无横向溢出和 console error。
- 新输出不含密钥、Token、Cookie、真实源 IP 或无关个人信息。
- 输出目录包含测试结果、失败记录、未验证项和 closeout。
- 只把已确认事实写入 `PROJECT_CONTEXT.md`、`TODO.md`、`DECISIONS.md`、`BUG_NOTES.md`。

只有通用验收门和专属验收门都通过，当前 Goal 才能标记为 `completed` 并进入下一项。

## 额度与恢复规则

- 如果 Codex 额度耗尽、客户端停止或上下文被压缩，不把 Goal 标记为失败或 blocked。
- 在机器状态中写入当前 Goal、最后完成检查点、最近通过的验证、未完成动作和 `waiting_for_quota`。
- 额度恢复后由同一任务的 heartbeat 读取状态并从最后 checkpoint 继续；不得重新执行已完成的真实模型调用、生产写入或大批量测试。
- 若无法可靠判断旧动作是否完成，先只读核对 Git、输出文件、进程、日志或线上状态，再决定是否重试。

## 用户确认门

遇到以下任一情况，状态改为 `waiting_for_user` 并停止：

- commit、push、部署、发布或 PR；
- 生产诊断 POST、生产 runtime 写入或生产数据迁移；
- API Key、IP 白名单、云权限、DNS、账号后台或验证码；
- 任何可能付费的模型、搜索、浏览器云服务或第三方 API；
- 数据库、登录、支付、Docker、队列或新基础设施；
- 产品定义、评分口径或证据合同发生根本冲突；
- 需要删除、覆盖、重置或处理来源不明的工作区文件。

等待时只通知一次相同阻塞；状态未变化时返回 `DONT_NOTIFY`。用户确认后，把确认范围和时间写入状态文件，再继续对应动作。

## G1：实体判定器与采样前来源核验

### Outcome

让系统能基于品牌、营销名、产品名、官方型号、城市和门店层级正确判断实体；在用户输入与已核验来源冲突时，先提示确认或修正，再进入昂贵的四模型采样。

### 输入

- `outputs/h5-mvp/source-recognition-hardening-20260712/`
- `outputs/h5-mvp/report-credibility-hardening-20260712/`
- `outputs/h5-mvp/batch-instance-testing-20260712/`
- 当前 PageAudit、preflight、credibility、rules 和报告生成代码及测试。

### 范围

- 建立通用的实体 alias/层级表示，至少覆盖品牌、营销名、产品名、主型号、城市、门店。
- 校准答案判定：不能只依赖完整目标字符串，也不能因为出现品牌名就自动算正确识别。
- 明确区分页面主实体、同类产品、推荐区域和用户填写事实。
- 把来源/型号/范围冲突检查放到真实 provider 采样之前；冲突时返回可理解的确认状态，并证明不会消耗模型额度或诊断限额。
- 拆分“提交来源可信度”和“站点基础设施完整度”的展示/数据语义，避免官方产品页有效却被聚合低分误解为来源不可信。

### Out of scope

- 不改公开营销内容，不做整体 UI 重设计，不调用真实 provider，不部署。

### 专属验收

- 固定人工标注 benchmark 覆盖 P01、P02、S01、S02、L01、L03、无关 URL、相关产品干扰和品牌简称。
- 至少提供一个“旧规则误判、修复后通过”的 before/after 例子。
- ES-LM55 页面不能把 ES-LV9C 相关产品区域当成主型号。
- 回答使用合理简称时可判定实体，但仅复述品牌仍不算自然推荐。
- 用户输入 ES-LV9C、来源主型号 ES-LM55 时，采样前停止并返回事实冲突；真实模型调用和限额消耗均为 0。
- 来源可信度与站点基建完整度在 H5/Markdown/HTML/evidence package 中语义一致。

### 输出

`outputs/h5-mvp/optimization-loop/g1-entity-evaluator/`

## G2：AI曝光体检官方内容与结构化证据

### Outcome

让公开页面对“AI曝光体检是什么、不是什么、适合谁、如何工作、证据与隐私边界”形成一致且机器可读的官方定义，降低模型把它误解成医学体检、隐私曝光或广告曝光的风险。

### 输入

- G1 closeout 和新的实体/来源契约。
- 当前 H5 首页、独立介绍页、隐私政策、用户协议、FAQ/功能入口、robots、sitemap、llms 文件和静态元数据。
- S01 真实四模型报告：正确实体识别 5%、无品牌词自然推荐 0%、10 组冲突。

### 范围

- 统一产品一句话定义、类别词、目标用户、工作流程、输入输出和能力边界。
- 明确它不是医学体检、身体/影像检测、隐私泄露扫描或广告投放曝光统计工具。
- 补齐功能说明、FAQ、隐私和数据处理说明、评分解释、证据边界、机器可读入口与必要结构化数据。
- 所有页面保持相同产品名、类别、主体和核心事实；不得编造客户案例、媒体背书、效果提升或排名承诺。
- 提供无品牌词真实问题的内容入口，但不把发布内容本身宣称为已获得模型收录。

### Out of scope

- 不做外部媒体投稿、付费推广、跨平台发布、真实模型复测或整体视觉重做。

### 专属验收

- PageAudit 能命中产品定义、类别、目标用户、核心流程、隐私边界和评分边界。
- 首页、介绍页、FAQ、隐私页、`llms.txt`/等效机器入口不存在相互冲突的定义。
- 至少提供医学体检误解的 before/after 页面文案例子。
- SEO/结构化数据通过本地解析；robots、sitemap 和 canonical 保持正确。
- 双端浏览器验证通过；无虚构事实或无法证明的业务效果。

### 输出

`outputs/h5-mvp/optimization-loop/g2-official-content-evidence/`

## G3：业务类型 Prompt、PageAudit 与来源安全体系

### Outcome

让实体商品、软件/小程序和本地服务使用符合真实决策过程的问题与来源规则，同时增强 PageAudit 的动态页面处理、事实溯源、时效与安全边界。

### 输入

- G1/G2 closeout。
- 20 例 batch matrix、L01/L03/L04、P01/P02、S01/S02 的历史证据。
- 当前 Prompt Universe、PageAudit fetch/parse、来源映射和安全校验代码。

### 范围

- 本地服务问题改为距离、门店、营业时间、排队、预约、菜单/服务范围、价格口径、停车、过敏与儿童服务；避免把消费者问题误导成加盟/企业采购合同问题。
- 商品问题围绕主型号、规格、安全标准、购买渠道、耗材、保修和同口径比较。
- 软件问题围绕功能、输入输出、隐私、主体、适用人群、限制和替代方案。
- 为需要 JavaScript 的官方页面设计受控渲染 fallback；渲染成功不自动升级为具体城市/门店事实。
- 增加 SSRF/私网地址、重定向、DNS rebinding、响应大小、MIME、超时和循环跳转防护测试。
- 为事实记录 URL、标题、canonical、命中片段/字段、抓取时间和内容哈希；区分当前、可能过期和失效证据。
- 候选来源发现只允许设计本地接口和审核状态；不接入新搜索 provider，不把搜索候选直接当官方来源。

### Out of scope

- 不修改云白名单，不引入付费搜索，不建立数据库，不抓取登录页或导入浏览器 Cookie。

### 专属验收

- 表驱动测试证明三类业务问题不串类。
- L01 能识别官方入口但仍要求具体西安门店证据；L03 全国首页继续被判 scope mismatched。
- L04 的价格、效果、安全和资质承诺继续留在待核验。
- 动态渲染、静态抓取和安全拒绝路径都有固定测试；安全用例不能发起对私网资源的真实访问。
- H5 与三类导出展示相同来源、时效、范围和暂停评分原因。

### 输出

`outputs/h5-mvp/optimization-loop/g3-domain-prompt-source/`

## G4：生产 C4、额度保护与运行观测

### Outcome

先在 L2 范围完成生产候选版本的可靠性、额度保护、延迟与观测验证；获得用户单独 L3/L4 和生产数据授权后，再提交、推送、部署并用一份受控诊断证明真实用户核心流程达到 C4。

### 输入

- G1-G3 closeout 和所有回归。
- `apps/ai-exposure-check-h5/docs/production-runbook.md`
- 当前 provider health、限流、幂等、runtime、日志和部署结构。

### 范围

- 区分“配置 ready”“最近真实成功”和“当前免费/计费状态未知”，不得把健康检查包装成真实采样成功。
- 记录 provider/模型成功率、失败类型、fallback、P50/P95 延迟、最慢问题、PageAudit 和总耗时，不记录敏感输入。
- 修正与真实延迟不符的等待时间文案。
- 为腾讯/火山等无法机器证明 free-only 的 provider 增加 fail-close 操作口径、人工巡检点和可单独关闭开关；不得猜测余额。
- 在现有单实例边界内改进限流持久性和重启恢复；不引入数据库或多实例协调。
- 建立发布前 secret/bundle/evidence 扫描、release precheck、回滚和只读 smoke 清单。

### 强制确认点

本地 C2 候选版本完成后必须停止并请求用户确认以下具体范围：

1. 是否允许 commit 和 push（L3/L4）。
2. 是否允许部署现有 H5 target（L4）。
3. 是否允许一次生产诊断 POST、写入一份生产报告并消耗默认四模型调用（L4 + 生产数据/成本确认）。
4. 若控制台额度或云配置需要人工确认，列出具体后台和只读检查项，等待用户完成或授权。

### 专属验收

- L2 阶段：typecheck/test/build/diff、故障注入、重启恢复、限流、fallback、日志脱敏、延迟指标和双端 UI 全部通过。
- L4 获批后：release precheck 通过，部署可回滚，systemd active，公开路由与导出 smoke 通过。
- 只允许生成一份新的生产诊断；首次生成、同 ID 恢复、报告查询与导出成功，不重复消耗模型调用。
- 四 provider 的实际成功/失败和 fallback 必须如实记录；若任一成本/权限/额度边界不清，停止而不是伪造 C4。
- 最终明确 C2/C3/C4，不能用部署成功替代真实用户流程成功。

### 输出

`outputs/h5-mvp/optimization-loop/g4-production-c4-operations/`

- G4 达到 C4 后，总状态改为 `completed`，更新项目状态文档并停用 heartbeat 自动化。

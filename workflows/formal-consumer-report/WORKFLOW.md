# 正式 AI 可见度详细报告工作流

## 调用方式

用户可以直接说：

> 执行正式 AI 可见度详细报告：{实体名称}。资料：{官网、品类、目标用户、核心问题、竞品和地区}。

一次调用只处理一个实体、一个日期窗口和一份最终报告。用户明确要求“执行正式报告”即授权 Codex 在已登录的四个消费端中提交本协议内的问题并在项目 `outputs/` 写入证据；不授权购买、充值、改账号、导入 Cookie、绕过验证码、部署、发布或对外联系。

## 与免费 H5 的边界

- 免费 H5：四模型 API + 后端公开网页候选搜索，生成“初步诊断分”，用于发现问题和引导咨询。
- 正式报告：DeepSeek、通义、腾讯元宝、豆包消费端的独立新会话实测 + 公开来源逐条核验 + 人工评分。
- H5 的 API 回答和初步诊断分可以作为背景材料，但不能替代消费端证据，也不能直接进入正式报告结论。
- 消费端单次回答只代表本次、该账号、该模式和该时间窗口，不得表述为稳定排名。

## 输入要求

开始前复制并填写 `intake-template.json`，至少确认：

- 实体名称、业务类型、品类、核心问题、目标用户；
- 一个合理竞品；
- 官方 URL 或明确记录无官方 URL；
- 地区和语言；
- 客户最关心的 1—3 个决策问题；
- 是否允许在已登录消费端提交最多 40 次独立会话问题。

资料不足会改变实体身份或竞品选择时先停下澄清；只缺少可选补充时记录为待核验并继续。

## 单次运行目录

创建：

`outputs/customer-reports/YYYYMMDD-<entity-slug>/`

必须包含：

- `protocol.md`：本次时间窗、平台、模式、问题、停止条件；
- `intake.json`：冻结后的实体资料；
- `prompts.json`：逐字冻结的所有提示词；
- `run-state.json`：每个单元 pending / complete / failed / missing；
- `raw/consumer/<platform>/`：完整回答 JSON 与 Markdown；
- `screenshots/<platform>/`：对应截图；
- `raw/public-web/`：公开网页读取结果；
- `sources/source-map.csv`：主张、URL、来源类型、支持程度和核验状态；
- `scores.csv`：逐条回答评分；
- `report.md`：客户可读的完整报告；
- `decision.md`：内部质量判定、异常、是否可交付；
- `run-log.md`：时间、失败、重试和人工判断记录。

原始回答和截图只追加，不覆盖；评分修改要在 `run-log.md` 说明原因。

## 提示词与规模

1. 从 `prompts-template.json` 生成 4 条固定核心问题，逐字保持一致。
2. 根据品类生成最多 4 条扩展问题，覆盖使用场景、限制风险、价格/获取方式和关键事实。
3. 加入正对照 P1 和负对照 P1，用于确认当前平台模式没有整体失效或系统性编造。
4. 每个平台最多 10 条独立会话，四个平台合计最多 40 次消费端交互。
5. P2 和任何自然推荐问题绝不能出现目标品牌名。
6. 所有问题在执行前写入 `prompts.json` 并冻结；运行中不临时改词来“修正”结果。

## 消费端采集协议

- 平台：DeepSeek、通义、腾讯元宝、豆包。
- 尽量在同一 24 小时窗口完成。
- 每条问题使用全新会话；不追问、不补背景、不纠正。
- 记录日期、网页/App、可见模型或模式、是否自动联网、是否显示搜索过程、是否给出引用。
- 保存完整最终回答、链接列表、页面异常和截图。
- 不手动开启特殊搜索或深度模式，除非本次协议明确把该模式列为单独测试组。
- 不导入 Cookie，不绕过登录或验证码。没有登录态就记为 missing，并继续能执行的平台。
- 页面卡住、只回显提示词或请求失败时保留现场；同一单元最多刷新或重开一次，不能反复抽样直到得到满意答案。

## 公开来源核验

1. 优先核验官方站、官方帮助中心、应用商店、监管/工商公开页和一手发布。
2. 对回答中的核心产品定义、功能、限制、价格、主体和数据主张逐条建立 source-map。
3. 链接能打开不等于支持主张；必须标记 `supports / partial / does_not_support / inaccessible`。
4. 搜索摘要和候选 URL 只能是线索，读取页面后才能标记为 `verified_external`。
5. 不复制长段版权内容；只保留必要短摘录和自己的核验说明。

## 评分与护栏

逐条字段使用 `scores-template.csv`。报告层指标为：

- 品牌认知方向：30%；
- 无品牌自然提及与位置：25%；
- 核心事实与引用有效性：25%；
- 四平台方向一致性：10%；
- 严重编造护栏：10%。

总分只是“本次消费端证据得分”，不是稳定排名。出现以下任一情况时暂停展示总分，只展示事实：

- 可评分最终回答少于计划单元的 75%；
- 正对照在两个及以上平台无法识别；
- 负对照在两个及以上平台被编造成真实实体；
- 关键主张缺少可核验官方或可信来源；
- 截图、原始回答与评分行无法一一对应。

严重编造必须单列，不能被其他高分抵消。

## 执行顺序

1. 读取本文件和所有模板。
2. 冻结 intake、实体身份、竞品、正负对照和提示词。
3. 创建运行目录和 `run-state.json`，报告计划交互数与外部账号边界。
4. 检查四平台当前登录态和可见模式；不提交问题。
5. 按平台逐条使用新会话采集，立即保存原文和截图并更新状态。
6. 采集完成后核验公开来源，建立 source-map。
7. 逐条评分并独立复算汇总指标。
8. 按 `report-template.md` 生成报告，再做一次证据路径、分母、反例和链接支持度检查。
9. 在 `decision.md` 标记：可交付 / 带保留可交付 / 不可交付。
10. 只向用户交付报告和证据路径；未经明确授权不发送给客户、不部署、不发布。

## 停止条件

- 登录墙、验证码或账号风控无法正常通过；
- 同一平台连续 3 个单元失败且没有新证据；
- 页面行为或条款显示自动化不再可用；
- 实体或竞品身份存在歧义；
- 意外收费、购买提示或权限升级；
- 达到本次 40 次消费端交互上限。

停止不等于删除结果。保留已经取得的原始证据，记录缺口并判断是否还能形成带保留报告。
